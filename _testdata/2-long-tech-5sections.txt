从零构建一个 RAG 问答系统：技术选型、踩坑与优化全记录

RAG（Retrieval-Augmented Generation）是目前大模型落地最成熟的方案之一。它解决的问题很直接：大模型的知识有截止日期，且无法访问企业内部文档，而 RAG 通过在推理前检索相关文档片段，将外部知识注入 prompt，让模型基于实时、私有的上下文生成答案。

但"跑通 demo"和"生产可用"之间的鸿沟，远比想象中大。本文记录我从零搭建一个内部知识库问答系统的全过程，涵盖技术选型、工程实现、以及那些文档里不会写的坑。

第一步是文档解析。我们面对的不是规整的 Markdown 文件，而是 PDF、Word、甚至扫描件。我的建议是先上 PyMuPDF（fitz）处理 PDF，它比 pdfplumber 快一个数量级；Word 用 python-docx 足够；扫描件则需要引入 OCR，PaddleOCR 的中文识别比 Tesseract 好不少。但这只是开始。真实文档的排版千奇百怪：双栏、表格嵌套、页眉页脚混入正文、列表编号断裂……这些问题没有银弹，只能靠规则 + 正则一点点处理。我的经验是：把解析结果统一转成 Markdown 格式，后续下游处理统一用 Markdown 作为中间表示，能省去大量适配工作。

然后是分块策略。最简单的方案是按固定窗口切分（比如 512 token 一块，重叠 50 token），但这对长文档问题很大——一个函数的定义和它的调用示例可能被切在不同块里，检索时上下文断裂。更好的做法是按文档结构切分：对 Markdown 按标题层级建树，对代码按函数边界切分。我还加了一层语义分块：在结构切分的基础上，用一个小模型判断相邻两个句子是否属于同一语义单元，决定是否在此处切分。这块开销很小但效果提升明显。

向量化存储是整个系统的核心。我们用的是 BGE-large-zh 做 embedding，Milvus 做向量数据库。BGE 在中文场景的表现稳定，尤其在短文本检索上。但有个坑：query 的 embedding 和 document 的 embedding 不应该用完全相同的处理方式。文档侧保持原样，查询侧可以适当加一些假设性改写（比如用 LLM 把用户口语化的提问改写成文档中可能出现的表述），再去检索，召回率能提升 5-10 个百分点。

检索策略上，最容易被忽视的是混合检索。纯向量检索对关键词匹配不敏感——用户搜"K8s"可能匹配不到"Kubernetes"，但 BM25 索引能完美补上。成熟方案是 hybrid search：向量 + BM25 同时检索，把两边结果做融合排序（RRF 是最常用的算法，参数别调太多，默认 60 就很好）。另外多路召回也是个好习惯：把原查询拆成子问题分别检索，有时能带出单独一路检索漏掉的相关内容。

最后的生成环节，模板非常重要。一个合格的 RAG prompt 至少需要：明确告诉模型"只能用提供的上下文回答，不知道就说不知道"；把来源文档的标题或编号附在引用后面；如果用户问题与检索结果明显无关，允许模型拒绝回答。我们在 prompt 里还加了一条：回答末尾附上引用片段编号，方便用户回溯原始文档验证事实——这个细节在用户信任度上帮助很大。

上线后的持续优化同样不能忽视。我们做了三件事：一是记录每次查询的检索结果和用户反馈，用负反馈对做 hard negative mining 来微调 embedding 模型；二是对高频问题的答案做缓存，命中率超过 30%，延迟从 3 秒降到 200 毫秒；三是监控 embedding 调用的 P99 延迟和 Milvus 的索引内存占用，避免在流量上来后突然爆雷。做了一年下来最深的感触是：RAG 的难点不在模型，而在工程——那些看似无聊的文档解析、分块策略、检索融合，才是真正决定用户体验的地方。
